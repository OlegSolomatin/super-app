## 1. Архитектура Anti-Strategies Engine

### 1.1 Ключевые принципы

| Принцип | Описание |
|---------|----------|
| **Изоляция** | Каждая стратегия — отдельный движок в своей директории со своей БД |
| **Независимость** | Стратегии не используют общие шины/очереди — только свои потоки данных |
| **Сравнимость** | Единый формат логов и метрик для кросс-анализа |
| **Минимализм** | Минимум зависимостей от основного super-app бэкенда |

### 1.2 Структура директорий

```
backend/
  anti_engines/                   # Корень всех anti-стратегий
    fakeout_hunter/               # Стратегия 1
      __init__.py
      engine.py                   # Движок
      config.yaml                 # Настройки стратегии
      schema.sql                  # Схема БД (SQLite)
      run.py                      # Точка запуска (standalone процесс)
    anti_mm/                      # Стратегия 2
      ...
    news_contrarian/              # Стратегия 3
      ...
    rebate_miner/                 # Стратегия 4
      ...
    boredom_trader/               # Стратегия 5
      ...
    ratio_arb/                    # Стратегия 6
      ...
    shared/                       # Общие модули (только утилиты!)
      __init__.py
      binance_stream.py          # WS-клиент (копируется, не импортируется)
      logger.py                   # Форматированный логгер
      metrics.py                  # Единые метрики
    launcher.py                   # Менеджер запуска всех стратегий

    anti_engines/                 # Базы данных (SQLite)
      fakeout_hunter.db
      anti_mm.db
      news_contrarian.db
      rebate_miner.db
      boredom_trader.db
      ratio_arb.db
```

### 1.3 Схема БД (единый шаблон)

Каждая стратегия имеет свою БД SQLite с единой структурой:

```sql
-- Мета-информация о запусках
CREATE TABLE runs (
    id TEXT PRIMARY KEY,           -- UUID
    strategy TEXT NOT NULL,        -- Название стратегии
    pair TEXT NOT NULL,            -- Торговая пара
    started_at TEXT NOT NULL,      -- ISO timestamp
    ended_at TEXT,                 -- ISO timestamp
    status TEXT NOT NULL DEFAULT 'running',  -- running / stopped / error / completed
    virtual_balance REAL NOT NULL DEFAULT 1000.0,
    config_json TEXT               -- JSON с конфигурацией запуска
);

-- Сигналы (каждое решение стратегии)
CREATE TABLE signals (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id TEXT NOT NULL REFERENCES runs(id),
    timestamp TEXT NOT NULL,       -- ISO timestamp
    type TEXT NOT NULL,            -- entry / exit / cancel
    direction TEXT,                -- long / short
    price REAL,                    -- Цена в момент сигнала
    reason TEXT,                   -- Почему сигнал сработал
    metadata_json TEXT,            -- Доп. данные (JSON)
    executed INTEGER DEFAULT 0     -- 0 = не исполнен (тест), 1 = исполнен
);

-- Сделки (исполненные сигналы)
CREATE TABLE trades (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id TEXT NOT NULL REFERENCES runs(id),
    signal_id INTEGER REFERENCES signals(id),
    entry_time TEXT NOT NULL,
    exit_time TEXT,
    entry_price REAL NOT NULL,
    exit_price REAL,
    direction TEXT NOT NULL,
    quantity REAL NOT NULL,
    pnl REAL,                      -- Прибыль/убыток
    pnl_pct REAL,                  -- В процентах
    fee REAL DEFAULT 0,
    status TEXT DEFAULT 'open'     -- open / closed / cancelled
);

-- Метрики производительности (обновляются каждые N секунд)
CREATE TABLE metrics_snapshots (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id TEXT NOT NULL REFERENCES runs(id),
    timestamp TEXT NOT NULL,
    balance REAL,
    total_signals INTEGER,
    executed_trades INTEGER,
    win_trades INTEGER,
    loss_trades INTEGER,
    current_drawdown_pct REAL,
    signals_per_minute REAL,
    additional_json TEXT
);

-- Индексы для быстрых запросов
CREATE INDEX idx_signals_run ON signals(run_id);
CREATE INDEX idx_trades_run ON trades(run_id);
CREATE INDEX idx_metrics_run ON metrics_snapshots(run_id);
```

### 1.4 Единый формат метрик (сравнительный анализ)

Каждый запуск любой стратегии генерирует одинаковый набор KPI:

| Метрика | Формат | Назначение |
|---------|--------|-----------|
| ROI | % | Общая доходность за запуск |
| WinRate | % | Доля прибыльных сделок |
| Profit Factor | число | Валовая прибыль / валовый убыток |
| Max Drawdown | % | Максимальная просадка |
| Sharpe Ratio | число | Доходность / риск |
| Avg Trade Duration | секунды | Среднее время удержания |
| Avg Win / Avg Loss | число | Соотношение средней прибыли к среднему убытку |
| Signals per Hour | число | Частота сигналов |
| Execution Rate | % | % сигналов, которые стали сделками |
| Calmar Ratio | число | ROI / Max DD |

### 1.5 Standalone запуск

Каждая стратегия запускается как **отдельный Python-процесс**:
- Независимый от uvicorn бэкенда
- Своё соединение с Binance WS
- Своя БД
- Останавливается через SIGTERM (graceful shutdown с сохранением состояния)

Управление через `launcher.py`:
```bash
# Запустить все стратегии в тестовом режиме (3 пробных запуска)
python -m backend.anti_engines.launcher --mode test

# Запустить одну стратегию
python -m backend.anti_engines.launcher --strategy fakeout_hunter

# Посмотреть статус всех запущенных
python -m backend.anti_engines.launcher --status

# Остановить все
python -m backend.anti_engines.launcher --stop-all

# Сравнительный отчёт по всем стратегиям
python -m backend.anti_engines.launcher --report
```

## 2. Стратегия 1: Fakeout Hunter (ложные пробои)

### 2.1 Концепция

> **«Против пробоя»** — цена пробивает уровень, но без реального объёма → высокая вероятность возврата.
> Вход ПРОТИВ пробоя, стоп за пробитым уровнем.

### 2.2 Триггеры входа (entry)

Стратегия анализирует depth stream в реальном времени:

```yaml
Условия для входа ПРОТИВ пробоя:
  - Цена пробила уровень N-тиков (N = от 3 до 10, конфиг)
  - Время пробоя: менее M миллисекунд (M = 500-2000ms)
  - Объём на пробое: меньше X% от среднего объёма кластера (X = 30-50%)
  - Отношение market maker / taker на пробое: < 1.5 (больше мейкеров = слабый пробой)
  - Глубина стакана с противоположной стороны: >= Y BTC на ближайших 5 уровнях
```

**Логика:**
- Средний объём кластера считается скользящим окном по последним N-свечам (на TFTick)
- Если пробой прошёл на 1-2 крупных market ордера — это стоп-охота, а не реальное движение
- Вход: limit ордер на 1-2 тика глубже пробитого уровня (лучшая цена)
- TP: возврат к средней кластера (50% глубины отката)
- SL: за пробитым уровнем + 1 тик (чтобы не выбило на ретесте)

### 2.3 Параметры стратегии

```yaml
fakeout_hunter:
  break_level_ticks: 5        # Сколько тиков считать «пробоем»
  break_time_ms: 1000         # Максимальное время пробоя (ms)
  volume_threshold_pct: 40    # Максимальный объём на пробое (% от среднего)
  maker_taker_ratio: 1.5      # Порог отношения мейкер/тейкер
  opposite_depth_btc: 2.0     # Минимальная глубина с противоположной стороны (BTC)
  take_profit_pct: 50         # % отката от пробоя до TP
  stop_loss_ticks: 2          # Тиков за пробой для SL
  max_concurrent_positions: 1 # Только 1 позиция
```

### 2.4 Когда НЕ торговать (фильтры отказа)

- Новостные окна (±5 минут от важных новостей)
- Периоды с VWAP-отклонением > 2% (сильный тренд)
- Если за последние 60 секунд было 3+ пробоя в одну сторону (усталость паттерна)
- Спред шире 0.05% (скользкий вход)

### 2.5 Риски

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Сильный тренд без отката | Средняя | VWAP-фильтр + стоп |
| Смена волатильности | Низкая | Фильтр ширина спреда |
| Большой игрок ломает паттерн | Низкая | Ограничение 1 поза |
| Ложный пробой → ещё один пробой | Высокая | Счётчик пробоев за 60с |

## 3. Стратегия 2: Anti-MM (анти-маркетмейкер)

### 3.1 Концепция

> **«Снятие ликвидности»** — крупный лимитный ордер исчезает (не по сделке) = кто-то убрал поддержку/сопротивление.
> Маркетмейкеры ставят лимитники, чтобы собрать спред. Их отмена — сильнейший сигнал.

### 3.2 Триггеры входа

Анализ depth stream на предмет **изменений** (delta между снапшотами):

```yaml
Условия для входа:
  - Крупный лимитный ордер (>= X BTC/$) БЫЛ на бид/аск
  - В следующем снапшоте (100ms) он ИСЧЕЗ без изменения цены последней сделки
  - Объём исчез: >= Y% от всей глубины на этом уровне
  - Противоположная сторона: не добавила сопоставимый объём (< 10% от снятого)
  - Время: не во время аукциона открытия/закрытия
```

**Логика:**
- Если с бида сняли 5 BTC — кто-то перестал поддерживать цену
- Если с аска сняли 5 BTC — кто-то убрал предложение
- Немедленный вход в направлении ОТ снятого ордера

**Детекция «снятия без сделки»:**
- Снапшот N: на биде 10.00 BTC по 50000.0 — 5.0 BTC
- Снапшот N+1: на биде 10.00 BTC по 50000.0 — 0 BTC
- Последняя сделка между N и N+1: не было сделок по 50000.0
- **Вывод:** лимитный ордер снят (не исполнен)

### 3.3 Параметры

```yaml
anti_mm:
  min_order_btc: 1.0            # Минимальный размер ордера для отслеживания
  min_removal_pct: 50           # % уровня который исчез
  opposite_add_threshold_pct: 10 # Максимум % добавленного с другой стороны
  snapshot_interval_ms: 100     # Интервал снапшотов
  position_size_pct: 20         # % от удалённого объёма как размер позиции
  take_profit_pct: 30           # % глубины возврата
  stop_loss_tick_offset: 3      # Тиков от уровня снятия
  max_daily_trades: 50          # Лимит сделок в день
```

### 3.4 Фильтры отказа

- Если противоположная сторона добавила сопоставимый объём — это ротация (не сигнал)
- Если цена ушла > 0.1% от точки снятия — опоздали, не входить
- Не входить в первые 30 минут после открытия сессии (ложные ордера)

### 3.5 Риски

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Ордер снят для перестановки выше/ниже | Высокая | Фильтр «противоположная сторона не добавила» |
| Ложное снятие (технический сбой) | Низкая | Проверка 2 снапшота |
| Iceberg ордер маскирует движение | Средняя | Следить за cumulative объёмом |

## 4. Стратегия 3: News Contrarian (новости назад)

### 4.1 Концепция

> **«Покупай на панике, продавай на эйфории»** — НО с задержкой в 15-30 минут, когда эмоции схлынули.
> Все бегут за движением → ты ждёшь и смотришь кто реально набирал позицию.

### 4.2 Архитектура

Отличается от остальных — **НЕ использует depth stream в реальном времени**.

```yaml
Режим работы:
  - Постоянно: парсинг новостного канала (Telegram / RSS / Twitter)
  - При обнаружении ключевого события: режим ожидания
  - Через T = 15 минут после события: анализ last N-свечей на объём
  - Решение: входить против направления первоначального движения
```

**Источники новостей:**
- Cryptopanic API (free tier)
- RSS ленты новостных сайтов
- Telegram каналы

**Анализ объёма после новости:**
```
1. Берём 3 свечи ДО новости → средний объём V_avg
2. Берём свечи ВО время новости (первые 2-3 минуты) → V_spike
3. Ждём 15 минут → V_after
4. Сравниваем:
   - Если V_after > V_avg × 1.5 И цена всё ещё в направлении первоначального движения:
     → Умные деньги продолжают набирать → входить в том же направлении
   - Если V_after ≈ V_avg И цена начала откатывать:
     → Паника улеглась → входить ПРОТИВ первоначального движения
   - Если V_after < V_avg И цена вернулась:
     → Фейковая новость → ничего не делать
```

### 4.3 Параметры

```yaml
news_contrarian:
  wait_minutes: 15              # Время ожидания после новости
  pre_event_candles: 3          # Свечей до события для среднего объёма
  spike_candles: 3              # Свечей во время события
  volume_surge_threshold: 2.0   # Порог всплеска объёма (× от среднего)
  contrarian_volume_threshold: 1.5  # Порог для contrarian входа
  position_size_usdt: 100       # Фиксированный размер позиции
  stop_loss_pct: 2.0            # Стоп-лосс от точки входа
  take_profit_pct: 3.0          # Тейк-профит
  max_open_positions: 1
```

### 4.4 Когда НЕ торговать

- Если движения нет (менее 0.5% за первые 5 минут) — новость слабая
- Если объём во время события меньше среднего — ложная тревога
- Если в ожидании появилась ещё одна новость — отмена текущего анализа

### 4.5 Риски

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Новость переворачивает рынок навсегда | Низкая | Стоп-лосс 2% |
| Пропуск движения | Высокая | Это НЕ стратегия для трендов |
| Ошибочная классификация новости | Средняя | Только проверенные источники |
| Опоздание входа | Средняя | Limit ордер по рынку |

## 5. Стратегия 4: Rebate Miner (анти-PnL)

### 5.1 Концепция

> **«Зарабатывай не на сделках, а на комиссиях»** — делай сотни «убыточных» микро-сделок получая maker rebate от биржи.
> 1-2 крупных движения перекрывают сотню микро-лоссов.

**Ключевая идея:** Maker ордера на Binance фьючерсах ДАЮТ rebate 0.01-0.02%. Если делать сотни тысяч сделок с мизерным убытком (-0.005% каждая) и собирать rebate (+0.015%), то на дистанции 0.01% × 100 000 сделок = профит.

### 5.2 Механизм

```yaml
Алгоритм:
  1. Ставим лимитный ордер на 1-2 тика от текущей цены (лучшая цена)
  2. Ждём когда его исполнят (мы мейкер — даём ликвидность)
  3. Сразу ставим обратный лимитный ордер (на 1-2 тика дальше)
  4. Если исполнился — профит от спреда + rebate × 2 (вход + выход)
  5. Если НЕ исполнился за N секунд — отменяем

Результат каждой сделки:
  - Прибыль от спреда: минимальная (0-1 тик)
  - Rebate: +0.015% за вход + 0.015% за выход = 0.03% гарантированных
  - Риск: движение против нас ломает спред
```

**Почему это НЕ обычный маркетмейкинг:**
- Нет анализа направления цены
- Нет прогнозирования
- Просто «сидим в спреде» на высоколиквидных парах
- Количество сделок = единственный фактор успеха

### 5.3 Параметры

```yaml
rebate_miner:
  tick_offset_entry: 1          # Тиков от лучшей цены для входа
  tick_offset_exit: 1           # Тиков для выхода
  order_timeout_seconds: 5      # Таймаут отмены ордера
  min_spread_usdt: 0.01         # Минимальный спред для входа
  max_position_usdt: 50         # Размер одной позиции
  max_concurrent_orders: 5      # Параллельных ордеров
  pair: "BTCUSDT"               # Только высоколиквидные
  min_daily_trades: 1000        # Цель сделок в день
  emergency_stop_drawdown: 5    # % просадки для аварийной остановки
```

### 5.4 Фильтры отказа

- Спред шире 0.02% → не входить
- Волатильность > 3% за 5 минут → пауза
- Новостное окно → пауза
- Скользящий PnL за час < 0 → сократить размер

### 5.5 Риски

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Проскальзывание при исполнении | Высокая | Только лимитные ордера |
| Скольжение рынка в одну сторону | Средняя | Выход по рынку при +3% движении |
| Binance отключает rebate | Низкая | Мониторинг условий |
| Технический сбой → зависшие ордера | Средняя | Таймаут отмены 5с |

## 6. Стратегия 5: Boredom Trader (скучный рынок)

### 6.1 Концепция

> **«Лучшая сделка — та, которую ты не совершил»** — но в данном случае: лучшая торговля — когда рынок спит.
> Торгуем ТОЛЬКО в периоды низкой волатильности (1-2% дневной диапазон).

**Логика:** в спокойные периоды:
- Спреды узкие → дешёвый вход/выход
- Маркетмейкеры активны → больше rebate (для мейкер-стратегий)
- Меньше «китов» которые двигают рынок
- Паттерны (поддержка/сопротивление) работают лучше
- Стоп-лоссы короче → меньший риск на сделку

### 6.2 Механизм

```yaml
Режим работы:
  1. Мониторинг волатильности (ATR на 1H, 5 min)
  2. Если ATR за последний час < порога → режим «скучно» → ВКЛ
  3. Если ATR > порога → режим «взрыв» → ВЫКЛ
  4. Во ВКЛ: простая стратегия с узкими стопами
  5. При первом признаке пробуждения (ATR > порога × 1.5) → закрыть всё

Стратегия во ВКЛ режиме:
  - Торгуем откаты от уровней (простая поддержка/сопротивление по price action)
  - Stoploss: 0.3% (короткий, так как рынок спокойный)
  - Take profit: 0.5-0.8%
  - Только 1 позиция
  - Только на парах с объёмом > 100M USDT/день
```

### 6.3 Параметры

```yaml
boredom_trader:
  atr_period_candles: 12         # Период ATR (1H свечи)
  atr_threshold_pct: 1.2        # % от цены — порог «скучно»
  atr_awake_multiplier: 1.5     # Множитель для выхода из режима

  take_profit_pct: 0.6          # Тейк в спокойном режиме
```

### 6.4 Фильтры отказа

- Волатильность > 1.2% за час → режим выключен полностью
- Объём пары < 100M USDT → слишком рискованно
- Время: 14:00-16:00 UTC (американская сессия, ложное спокойствие)
- Новостное окно (±30 минут)

### 6.5 Риски

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Внезапный выброс волатильности | Средняя | Мгновенное закрытие всех позиций |

| Разрыв (gap) при открытии | Низкая | Только фьючерсы (без гепов) |

## 7. Стратегия 6: Ratio Arb (индекс наоборот)

### 7.1 Концепция

> **«Не торговать пару — торговать разницу»** — открываем 2 позиции: лонг одного актива + шорт другого.
> Зарабатываем на возврате соотношения к среднему.

**Почему «индекс наоборот»:** вместо прогноза направления цены — прогноз соотношения между двумя коррелирующими инструментами.

### 7.2 Пары для Ratio Arb

```yaml
ratio_pairs:
  - name: "ETHBTC"
    base: "ETHUSDT"           # Лонг
    quote: "BTCUSDT"          # Шорт
    lookback_days: 30         # Период расчёта среднего
    entry_deviation: 0.15     # % отклонения от среднего для входа
    stop_loss_deviation: 0.5  # % отклонения для стопа
    position_ratio: 0.5       # Размер позиции от стандартного

  - name: "BNBBTC"
    base: "BNBUSDT"
    quote: "BTCUSDT"
    lookback_days: 30
    entry_deviation: 0.12
    stop_loss_deviation: 0.4
    position_ratio: 0.3

  - name: "SOLBTC"
    base: "SOLUSDT"
    quote: "BTCUSDT"
    lookback_days: 20
    entry_deviation: 0.2
    stop_loss_deviation: 0.6
    position_ratio: 0.25
```

### 7.3 Механизм

```yaml
Алгоритм:
  1. Каждые N минут рассчитываем соотношение: P_base / P_quote
  2. Скользящее среднее за lookback_days + стандартное отклонение
  3. Если ratio отклонился от среднего на entry_deviation × σ:
     - Если ratio ниже среднего → лонг base + шорт quote (надежда на возврат вверх)
     - Если ratio выше среднего → шорт base + лонг quote (надежда на возврат вниз)
  4. Размер: 50% портфеля в base, эквивалент в quote (нейтрально по BTC)
  5. Выход: ratio вернулся к среднему ± 0.5σ ИЛИ stop_loss_deviation
```

**Нейтральность по BTC:**
- Если лонг ETH на $100 + шорт BTC на эквивалент $100 → движение BTC не влияет на PnL
- Зарабатываем только на изменении соотношения

### 7.4 Параметры

```yaml
ratio_arb:
  rebalance_interval_minutes: 5   # Как часто пересчитывать ratio
  lookback_days: 30               # Период для среднего и σ
  entry_deviation_sigma: 2.0      # Сколько σ для входа (2 = 95% доверительный)
  exit_deviation_sigma: 0.5       # Сколько σ для выхода
  stop_deviation_sigma: 3.0       # Аварийный стоп
  base_position_usdt: 100         # Размер одной ноги
  max_open_pairs: 2               # Максимум одновременных ratio-позиций
  min_correlation: 0.7            # Минимальная корреляция для пары
```

### 7.5 Фильтры отказа

- Корреляция < 0.7 за lookback → пара не торгуется
- Если одна нога имеет ликвидность < 50 BTC → не входить
- Если ratio изменился > 3σ за 1 час → экстренный выход

### 7.6 Риски

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Декорреляция (корреляция падает) | Средняя | Пересчёт каждые 5 минут + min_correlation |
| Фандинг на фьючерсах | Средняя | Учитывать в расчёте PnL |
| Размер позиции слишком мал | Низкая | Увеличиваем при подтверждении стратегии |
| 3-й фактор влияет на оба актива | Высокая | Невозможно полностью захеджировать |

## 8. Стратегия 7: Short The Hype (мемкоин-шорт)

### 8.1 Концепция

> **«Покупай слухи — продавай факты» НО наоборот: пропускай пампа, шорть после эйфории.**
> Мемкоин выстрелил +100-1000% за сутки → ждём пока хайп схлынет → шортим с новостным фильтром.

**Почему это работает:** мемкоины живут на хайпе. После первого взрывного роста:
- Ранние зашедшие фиксируют прибыль (dump)
- Новые покупатели на пике остаются в минусе
- Если нет новых позитивных новостей → цена возвращается на 50-80% от пика

**Почему НЕ делает никто:** все хотят зайти в пампающийся токен на взлёте. Шортить мемкоин, который только что вырос в 10 раз, страшно — кажется что «он может выстрелить ещё». Но в 90% случаев после взрывного роста следует глубокая коррекция.

### 8.2 Механизм

```
Фаза 1: ОБНАРУЖЕНИЕ пампа
  └→ Мониторинг топ-100 мемкоинов по объёму (CoinGecko/CoinMarketCap API)
  └→ Триггер: рост > X% за Y часов при объёме > Z
  └→ Фиксация: токен, пиковая цена, время пика

Фаза 2: ОЖИДАНИЕ (cooldown)
  └→ Ждём T часов после пика (настраиваемый cool-down)
  └→ За это время: собираем новости по токену
  └→ Если за время ожидания появились сильные позитивные новости
      (листинг на Binance, партнёрство, Илон твитнул) → ОТМЕНА

Фаза 3: АНАЛИЗ перед входом
  └→ Текущая цена: упала ли на N% от пика?
  └→ Объём: снизился ли до < M% от пикового?
  └→ Новости: нет позитивных за последние 6 часов?
  └→ Соцсети: активность падает? (опционально)
  └→ Если все условия соблюдены → вход

Фаза 4: ВХОД (short)
  └→ Размер: фиксированный % от виртуального баланса (например, 20%)
  └→ SL: выше пика на 10-15% (на случай второй волны)
  └→ TP1: 30% от пика к дну (частичный)
  └→ TP2: 50% от пика к дну
  └→ TP3: 70% от пика к дну (полный выход)

Фаза 5: МОНИТОРИНГ в позиции
  └→ Каждые N минут: проверка новостей по токену
  └→ Если вышла позитивная новость → немедленный выход
  └→ Если объём снова растёт → сократить позицию на 50%
```

### 8.3 Детекция пампа (триггеры)

```yaml
pump_detection:
  price_increase_pct: 100       # Минимальный рост для триггера (%)
  price_window_hours: 24        # За сколько часов
  volume_multiplier: 5          # Объём должен быть > 5× от среднедневного
  min_volume_usdt: 1_000_000    # Минимальный объём для рассмотрения
  max_market_cap_usdt: 5_000_000_000  # Не шортим «тяжёлые» монеты (DOGE, SHIB)
```

**Почему макс. капитализация:** мемкоины с капитализацией >$5B (DOGE, SHIB) уже слишком «тяжёлые» для взрывных движений — их рост/падение растянуто во времени. Стратегия работает на средне-мелких мемкоинах (PEPE, WIF, BONK, TRUMP, и т.д.)

**Откуда брать данные:**
```yaml
data_sources:
  - CoinGecko API (бесплатно, 50 запросов/мин)
    endpoint: /coins/markets?vs_currency=usd&order=volume_desc
  - Binance API (для тех что на Binance)
  - Jupiter API (для Solana-мемкоинов)
```

### 8.4 Новостной фильтр (ключевая особенность)

Отличает эту стратегию от простого «шортим всё что выросло».

```yaml
news_filter:
  check_interval_minutes: 15    # Проверка новостей каждые 15 мин
  sources:
    - cryptopanic_api           # Агрегатор крипто-новостей
    - twitter                    # Поиск по имени токена
    - telegram_channels          # Отслеживание каналов
    - cointelegraph              # Крупные СМИ
    - binance_announcements     # Листинги

  # Типы новостей по силе влияния
  strong_positive:              # → ОТМЕНА входа / выход из позиции
    - "listing"
    - "partnership"
    - "major exchange"
    - "Elon Musk"
    - "Trump"
    - "whale accumulation"
    - "burn mechanism"

  medium_positive:              # → Сократить размер / уменьшить TP
    - "community growth"
    - "new feature"
    - "marketing campaign"
    - "celebrity endorsement"

  negative:                     # → Увеличить размер позиции / ускорить вход
    - "rug pull"
    - "hack"
    - "insider selling"
    - "SEC investigation"
    - "founder drama"
```

**Логика фильтра:**
```
Обнаружен пампа на +200% → ждём 12 часов
Параллельно: каждые 15 минут проверяем новости

Сценарий А: новостей нет или негативные
  ✅ → шорт (хайп схлынул, возврат неизбежен)

Сценарий Б: новость "Binance Listing" 
  ❌ → ОТМЕНА (это начало, а не конец пампа)

Сценарий В: новость "celebrity tweet" (средняя)
  ⚠️ → шорт, но половинным размером (может быть вторая волна)
```
